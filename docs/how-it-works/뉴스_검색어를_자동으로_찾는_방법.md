# 뉴스 검색어를 자동으로 찾는 방법

> 보유 종목의 최신 뉴스를 놓치지 않기 위해 Kiwi와 TF-IDF로 확장 검색어를 자동 발견한 구현 이야기

## 1. 무엇을 만들었나

사용자가 보유한 종목의 회사명이 직접 들어간 기사뿐 아니라, 최근 해당 회사와 함께 자주 언급되는 제품·사업·이슈 기사까지 자동으로 찾는 뉴스 수집 MVP를 구현했다. 이 문서는 공통 구현 원리를 설명하고 삼성전자를 실제 검증 사례로 사용한다.

이 작업은 언론사 웹페이지의 HTML을 무단으로 긁는 방식이 아니다. 현재는 NAVER 뉴스 검색 API에서 제공하는 제목, 요약, 원문 링크, 발행 시각을 수집한다. 뉴스 공급처가 바뀌어도 분석 로직을 재사용할 수 있도록 NAVER 호출 부분은 source adapter로 분리했다.

전체 흐름은 다음과 같다.

```text
삼성전자 직접 검색
→ 기사 정규화·중복 제거
→ 회사명이 제목에 있는 고신뢰 기사 선별
→ Kiwi로 명사·복합명사·영문 기술어 추출
→ TF-IDF 기반 점수와 품질 조건으로 최근 핵심어 선정
→ 삼성전자 + 핵심어로 확장 검색
→ 기사-검색어-종목 관계 저장
```

예를 들어 최근 직접 기사에서 `로봇`, `갤럭시`, `AI`가 반복적으로 확인되면 다음 검색어를 자동으로 만든다.

```text
삼성전자 로봇
삼성전자 갤럭시
삼성전자 AI
```

`AI`나 `로봇`만 검색하면 다른 기업의 기사가 너무 많이 섞일 수 있기 때문에, 현재 MVP는 회사명을 함께 넣어 정밀도를 우선한다.

## 2. 왜 회사명만 검색하지 않았나

`삼성전자`만 검색해도 직접 언급 기사는 찾을 수 있지만 다음과 같은 산업·제품 이슈를 놓칠 수 있다.

- 갤럭시 신제품과 폴더블폰 시장 변화
- HBM, 메모리, 파운드리 관련 공급망 변화
- 로봇과 피지컬 AI 사업 확대
- 반도체 수출 규제나 공급 부족

반대로 `삼성`, `반도체`, `AI`를 무조건 단독 검색하면 삼성 계열사, 경쟁사, 증시 시황 등 무관한 기사가 대량으로 섞인다.

따라서 키워드를 미리 삼성전자 전용 목록으로 고정하지 않고, 최근 삼성전자 직접 기사에서 실제로 반복된 표현을 발견해 확장 검색어로 사용한다.

## 3. Kiwi는 어디에 사용했나

Kiwi는 한국어 문장을 형태소와 품사로 분석하는 도구다. NAVER가 제공한 기사 제목과 요약에서 다음 품사를 추출한다.

| Kiwi 품사 | 의미 | 예시 |
|---|---|---|
| `NNG` | 일반명사 | 로봇, 반도체, 가전 |
| `NNP` | 고유명사 | 갤럭시 |
| `SL` | 영문 표현 | AI, HBM, RX |

예를 들어 다음 제목이 있다고 하자.

```text
삼성전자, 갤럭시 글래스 출시…차세대 웨어러블 시장 공략
```

Kiwi 분석을 거치면 `삼성전자`, `갤럭시`, `글래스`, `출시`, `웨어러블`, `시장` 같은 명사 후보를 얻는다. 실제로 서로 인접한 명사는 `갤럭시 글래스`, `메모리 반도체`처럼 복합명사 후보로도 만든다.

이 단계의 역할은 **문장에서 검색어가 될 만한 표현을 찾아내는 것**이다. Kiwi가 중요도를 판단하는 것은 아니다.

### 불필요한 후보 제거

추출한 명사를 그대로 사용하면 품질이 낮아지므로 다음 표현은 제거하거나 단독 활성화를 막는다.

- 일반어: 회사, 관련, 시장, 기자, 전망, 발표, 상승, 하락
- 대상 회사명: 삼성전자
- 모호한 그룹명: 삼성, 현대, SK, LG
- 다른 지원 종목명: SK하이닉스, 현대차 등
- 범위가 너무 넓은 단독어: 반도체, 자동차, 금융 등
- 한 기사에만 우연히 등장한 표현

이 규칙은 삼성전자에만 적용되는 수동 라벨이 아니라 모든 지원 종목에 공통으로 적용된다.

## 4. TF-IDF는 어디에 사용했나

Kiwi가 만든 후보 중 최근 기사에서 상대적으로 중요한 표현을 고르기 위해 TF-IDF 개념을 사용했다.

현재 구현은 scikit-learn의 `TfidfVectorizer`를 그대로 호출하는 방식이 아니라, 뉴스 수집 목적에 맞게 제목 가중치와 언론사 다양성을 추가한 TF-IDF 기반 점수 계산이다.

### 4.1 문서 빈도와 IDF

한 단어가 몇 개 기사에 등장했는지 문서 빈도 `df`를 계산한다. 한 기사에서 같은 단어가 여러 번 반복돼도 문서 한 건으로 계산해 특정 기사의 반복 문구가 점수를 과도하게 높이지 않게 한다.

```text
IDF = log((전체 기사 수 + 1) / (해당 단어가 등장한 기사 수 + 1)) + 1
```

거의 모든 기사에 나오는 일반적인 단어보다 일부 관련 기사에서 집중적으로 반복되는 표현을 구분하기 위한 값이다. 단, 한 건에만 등장하는 희귀어가 과대평가되지 않도록 최소 기사 수 조건을 별도로 둔다.

### 4.2 제목 가중치

뉴스 제목은 해당 기사의 핵심을 압축한 부분이므로 요약보다 높은 가중치를 준다.

```text
요약에 등장: 1.0
제목에도 등장: 1.5 추가
제목 등장 시 합계: 2.5
```

### 4.3 언론사 다양성

한 언론사의 반복 기사보다 여러 언론사가 함께 다룬 주제를 더 신뢰한다.

```text
최종 점수
= 제목·요약 가중 빈도
× IDF
× (1 + 0.2 × (언론사 수 - 1))
```

따라서 여러 기사와 여러 언론사의 제목에서 반복된 `로봇`, `갤럭시` 같은 단어가 높은 점수를 받는다.

## 5. 검색어로 승격되는 조건

점수가 높아도 다음 조건을 모두 만족해야 실제 검색어가 된다.

- 회사명이 제목에 직접 등장한 고신뢰 기사에서 발견
- 서로 다른 기사 2건 이상 등장
- 서로 다른 언론사 2곳 이상 등장
- 제목에 최소 1회 등장
- 일반어, 모호한 그룹명, 경쟁사명이 아님
- 의미가 겹치는 키워드의 중복 활성화 방지
- 종목당 최대 5개만 활성화
- 발견 시점으로부터 7일 뒤의 만료 기준시각 기록

최근 로컬 실행에서 삼성전자 기사로부터 다음 활성 키워드가 선택됐다.

```text
로봇
AI
RX
가전
갤럭시
```

키워드 점수는 고정된 회사 카테고리가 아니라 실행 시점의 최근 기사 corpus에 따라 달라진다. 최근 기사에서 HBM 언급 근거가 부족하면 HBM을 억지로 넣지 않는다.

현재는 후보마다 `expires_at`을 기록하는 단계까지 구현했다. 만료 시각이 지난 후보를 자동 비활성화하고 새 후보로 교체하는 정기 작업은 운영 배치 단계에서 추가해야 한다.

## 6. 이벤트 조합은 어떻게 만드나

`HBM`과 `공급 부족`이 각각 발견됐다는 이유만으로 두 표현을 임의로 합치지 않는다. 같은 기사 안에서 두 표현이 실제로 함께 등장하는지 확인한다.

```text
기사 A: HBM 공급 부족 우려
기사 B: HBM 공급 부족 장기화
→ 삼성전자 HBM 공급 부족 검색어 생성 가능
```

이벤트 조합도 기사 2건 이상, 언론사 2곳 이상, 제목 등장 조건을 통과해야 활성화된다. 현재 확인하는 사건 표현에는 공급 부족, 공급 차질, 수출 규제, 판매 부진, 리콜, 관세, 증설, 감산, 수주, 출시, 양산 등이 있다.

## 7. 수집 결과는 어떻게 저장하나

기사 자체와 검색 근거를 분리해 저장한다.

```text
articles.jsonl
└─ 중복 제거된 공용 기사

article_queries.jsonl
└─ 어떤 검색어로 기사가 들어왔는지 기록

article_companies.jsonl
└─ 기사가 어떤 종목과 어떤 근거로 연결됐는지 기록
```

같은 기사가 `삼성전자`, `삼성전자 로봇`, `삼성전자 AI` 검색에서 반복돼도 기사 본문 정보는 한 번만 저장하고 검색어 관계만 여러 개 저장한다.

검색어별 checkpoint도 관리한다. API는 최신 기사 확인을 위해 다시 호출하지만 이전에 처리한 URL은 신규 기사로 중복 저장하지 않는다.

## 8. 실행 방법과 API 호출 수

```bash
# 1. 삼성전자 직접 뉴스 수집: NAVER API 1회
.venv/bin/python -m collector.jobs.collect_company_news \
  --stock-code 005930

# 2. 저장된 기사에서 키워드만 발견: NAVER API 호출 없음
.venv/bin/python -m collector.jobs.discover_company_queries \
  --stock-code 005930

# 3. 활성 키워드 최대 5개로 확장 수집: NAVER API 최대 5회
.venv/bin/python -m collector.jobs.discover_company_queries \
  --stock-code 005930 \
  --collect-active
```

삼성전자 한 종목의 전체 흐름은 `회사명 1회 + 확장 검색 최대 5회`, 즉 한 번 갱신할 때 최대 6회 호출한다. 각 호출은 최대 100개 결과를 받는다.

## 9. 현재 결과를 어떻게 해석해야 하나

현재 구현은 **관련 기사 후보를 자동 수집하고 관련도 v2로 정제하는 로컬 MVP**다. 회사명·검색 주제의 제목 등장은 가점하고 증시 종목 나열, 채용, 판촉, 수상, 혼합 뉴스 모음은 감점한다. `eligible`, `candidate`, `rejected` 중 `eligible` 기사만 BERTopic 입력 파일에 포함한다.

2026-07-22 로컬 검증에서는 삼성전자 누적 기사 383건 중 137건이 BERTopic 입력 대상으로 남았다. 이 수치는 정답률이 아니라 현재 규칙의 통과율이며, 종목과 기간을 늘려 false positive와 false negative를 계속 확인해야 한다.

수집된 결과를 사용자에게 그대로 노출하지 않는다. 후속 BERTopic 파이프라인에서 같은 사건을 묶고 이슈 중요도를 계산해 대표 이슈 Top 3를 선정한다.

- 동일 사건 기사 묶기
- Topic 내부 날짜별 Event 분리
- 최신성·출처 다양성·직접성 기반 이슈 랭킹
- 추출형 의미 기반 주요 이슈명 생성

과거 유사 Event와 KIS 일봉 연결은 현재 구현 범위에서 보류했다.

이번 구현의 의미는 **수동으로 삼성전자 키워드 목록을 관리하지 않고, 수업에서 배운 Kiwi와 TF-IDF를 활용해 최근 뉴스에서 검색 주제를 자동 발견하는 기반을 만들었다는 것**이다.

## 10. 한 줄 요약

> Kiwi로 삼성전자 최신 기사에서 명사와 기술어를 찾고, TF-IDF·제목 가중치·언론사 다양성으로 중요한 키워드를 골라 `삼성전자 + 키워드` 뉴스까지 자동 수집한다.
