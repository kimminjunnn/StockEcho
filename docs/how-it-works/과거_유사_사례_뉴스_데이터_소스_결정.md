# 과거 유사 사례 뉴스 데이터는 어디서 가져올까

> BigKinds, NAVER, Google, Yahoo, GDELT, 크롤링과 OpenDART를 검토한 뒤 StockEcho MVP의 데이터 소스를 결정한 과정

## 1. 결정

StockEcho MVP의 과거 유사 사례 분석은 **NAVER API HUB 뉴스 검색 API를 주 데이터 소스로 사용한다.**

이 결정이 NAVER가 완전한 과거 뉴스 아카이브라는 뜻은 아니다. NAVER는 날짜 범위를 지정할 수 없고 검색어별로 접근 가능한 결과 위치가 최대 1,000이므로, 과거 전체 기사를 빠짐없이 확보할 수 없다. MVP에서는 이 한계를 받아들이고 다음 두 경로를 함께 사용한다.

```text
정기 수집
→ sort=date로 최신 뉴스를 계속 Supabase에 축적

과거 유사 사례 요청
→ 현재 Topic/Event 핵심어로 sort=sim 검색
→ 자사 → 공통 산업 → 동종기업 순서로 후보 확장
→ 관련도 평가, 날짜·출처 기준 Event 생성
→ 검색 결과를 캐시
```

사용자 화면에는 전체 언론 아카이브를 분석한 것처럼 표현하지 않는다.

```text
권장 문구
수집된 뉴스 기반 유사 사례

안내 문구
NAVER 검색 및 StockEcho가 수집한 뉴스 범위에서 분석한 결과입니다.
전체 언론 보도를 포함하지 않을 수 있습니다.
```

결과가 부족할 때는 낮은 품질의 사례를 억지로 채우지 않고 `충분한 과거 사례 없음` 상태를 표시한다.

## 2. 만들려는 기능

사용자가 현재 주요 이슈 카드의 `과거 유사 이슈 보기`를 누르면 다음 정보를 보여준다.

```text
현재 Topic/Event
→ 핵심 기업·산업·사건 표현 추출
→ 과거 유사 뉴스 검색
→ 날짜와 출처가 겹치는 기사를 하나의 과거 Event로 묶기
→ 대표 기사와 유사 근거 표시
→ Event 시점 이후 D+1·D+5·D+15·D+30 거래일 수익률 표시
```

한 날짜의 Event는 기본적으로 서로 다른 출처가 2곳 이상일 때 채택한다. 자사 과거 사례를 먼저 찾고, 부족할 때 공통 산업 사례와 동종기업 사례로 확장한다.

## 3. NAVER를 선택한 이유

### 장점

- 공식 API라 검색결과 HTML 크롤링보다 운영·법적 위험이 낮다.
- 제목, 요약, 원문 URL, NAVER URL, 발행시각을 구조화된 JSON으로 제공한다.
- 국내 언론사 범위가 넓다.
- 현재 프로젝트에 API HUB client, 정규화, 중복 제거, 관련도 평가, Supabase 저장이 이미 구현돼 있다.
- 최신 뉴스 자동 수집과 과거 유사 사례 후보 검색을 같은 데이터 모델로 처리할 수 있다.

현재 구현은 다음 파일에 있다.

```text
collector/clients/naver.py
collector/sources/naver.py
collector/jobs/collect_news_query.py
collector/jobs/collect_company_news.py
collector/jobs/collect_and_schedule.py
collector/jobs/backfill_issue_news.py
collector/historical_news/planner.py
```

### 한계

- `sort=sim`이 기본 정확도순이고 `sort=date`는 최신순이다.
- 시작일과 종료일을 지정하는 파라미터가 없다.
- 한 번에 최대 100건을 받는다.
- `start`의 최댓값은 1,000이다.
- 같은 검색어를 반복 호출해도 1,001번째 이후 결과로 이동할 수 없다.
- 검색 결과 총 개수가 1,000보다 많아도 실제 접근 가능한 것은 상위 1,000건이다.
- 검색어별 기사량에 따라 도달 가능한 과거 범위가 크게 달라진다.

최대 1,000건을 모두 받으려면 다음과 같이 10회 호출한다.

```text
display=100, start=1
display=100, start=101
...
display=100, start=901
```

API 사용량은 반환 기사 수가 아니라 HTTP 요청 횟수로 계산한다.

## 4. 실제 NAVER 검색 테스트

2026년 7월 23일 API HUB에서 `sort=date`로 각 검색어의 상위 1,000건을 직접 조회했다.

| 검색어 | NAVER 표시 총 결과 | 접근한 결과 | 가장 오래된 결과 |
|---|---:|---:|---|
| 반도체 원재료 상승 | 22,666 | 1,000 | 2026-05-11 |
| 반도체 원자재 가격 상승 | 41,516 | 1,000 | 2026-06-10 |
| 반도체 원자재 | 111,712 | 1,000 | 2026-07-07 |

검색어가 넓을수록 결과가 많아져 최신 1,000건이 빨리 채워지므로 오히려 과거로 덜 이동했다.

`반도체 원재료 상승 2021`처럼 연도를 검색어에 추가하고 `sort=sim` 상위 100건을 확인하는 우회도 테스트했다.

| 검색 연도 | 실제 해당 연도 기사 |
|---:|---:|
| 2021 | 0 |
| 2022 | 0 |
| 2023 | 0 |
| 2024 | 0 |
| 2025 | 0 |

검색어에 연도를 넣는 방식은 날짜 범위 검색의 대체 수단으로 사용할 수 없다.

## 5. 과거 유사 사례 검색 전략

현재 구현된 `backfill_issue_news`는 필요한 사례 수가 채워질 때까지만 단계적으로 NAVER를 호출한다.

```bash
.venv/bin/python -m collector.jobs.backfill_issue_news \
  --stock-code 005930 \
  --keyword 반도체 \
  --keyword 수출 \
  --keyword 규제 \
  --before 2026-07-22
```

검색 순서는 다음과 같다.

```text
1. 자사 + 사건 표현
   예: 삼성전자 반도체 수출 규제

2. 기업명을 뺀 공통 산업 표현
   예: 반도체 수출 규제

3. 동종기업 + 사건 표현
   예: SK하이닉스 반도체 수출 규제
```

보수적인 동의어 변형도 함께 만든다.

```text
수출 규제 ↔ 수출 통제
원재료 ↔ 원자재
공급 부족 ↔ 공급난
파업 ↔ 노사분규
```

기본 호출 예산은 검색어당 최대 3페이지, 전체 최대 12회다. 동일한 종목·키워드·기준일 요청은 결과 파일을 재사용한다.

```text
data/processed/historical_search/issue_search_<hash>.json
```

### 검색 결과를 Event로 채택하는 조건

- 현재 Event 기준일보다 과거 기사여야 한다.
- 기사 관련도 평가를 통과해야 한다.
- 같은 날짜와 같은 종목의 기사를 Event 후보로 묶는다.
- 기본적으로 서로 다른 원문 도메인이 2개 이상이어야 한다.
- 동일 URL과 동일 문서는 한 번만 계산한다.
- 자사 Event를 우선하고 부족한 수만큼 외부 기업 Event를 사용한다.

### 추가로 구현할 품질 기준

- 현재 Event와 과거 Event의 임베딩 유사도
- 기업, 산업, 사건 행위, 원인 표현의 개별 일치 점수
- 유사도 최저 기준
- 같은 사건의 연속 보도를 한 Event 기간으로 합치는 규칙
- 출처 다양성과 기사 수를 이용한 신뢰도
- 검색어가 넓어 생긴 무관 기사 제거

## 6. 주가 반응 계산 기준

과거 Event의 기준일 이후 거래일 수익률을 계산한다.

```text
기준가: Event 기준 거래일 종가
D+1: 다음 1번째 거래일 종가
D+5: 다음 5번째 거래일 종가
D+15: 다음 15번째 거래일 종가
D+30: 다음 30번째 거래일 종가
```

주말·휴일에 발생한 Event는 다음 거래일을 기준일로 정한다. 기사 발행시각을 사용할 수 있으면 장 마감 전후를 구분하고, 시각이 불확실하면 미래정보 누출을 피하기 위해 다음 거래일을 기준으로 사용하는 보수적인 규칙을 적용한다.

수익률은 다음과 같이 계산한다.

```text
수익률(%) = (비교일 종가 / 기준가 - 1) × 100
```

거래정지, 상장 전, 상장폐지, 가격 데이터 누락은 정상 수익률로 간주하지 않고 별도 상태로 표시한다.

## 7. 검토했지만 채택하지 않은 데이터 소스

### BigKinds 유료 API

- 국내 뉴스 아카이브와 인물·기관·분류·키워드 정보가 우수하다.
- 정확한 과거 데이터 구축에는 가장 적합한 후보였다.
- 현재 예산에서 구매가격이 높아 MVP에서는 제외했다.

### BigKinds 수동 XLSX

실제 `NewsResult_20260423-20260723.xlsx`를 확인했다.

```text
언론사: 한국경제
행 수: 20,000
실제 날짜: 2026-05-27 ~ 2026-07-23
컬럼 수: 19
본문: 최대 200자
뉴스 식별자: 20,000개 모두 고유
```

요청 기간은 2026-04-23부터였지만 최대 20,000건 제한 때문에 앞부분이 잘렸다. 월별 다운로드와 수동 적재가 필요하고 지속 운영 부담이 커서 채택하지 않았다.

파일 내부 `dimension` 메타데이터가 `A1`로 잘못 기록돼 일반적인 XLSX read-only parser가 한 셀만 있는 파일로 오인하는 문제도 확인했다. 전용 importer를 만들면 해결 가능하지만, 수동 다운로드 의존성을 없애기로 결정했으므로 현재 구현 범위에서는 제외한다.

### Google

Programmable Search에는 날짜 범위 검색 문법이 있지만 뉴스 전용 아카이브가 아니다. 신규 고객용 JSON API가 닫혔고 기존 서비스도 종료 예정이라 새 핵심 의존성으로 부적합하다.

Google News 웹·RSS의 `after`, `before` 검색은 공식적으로 보장된 운영 API가 아니므로 사용하지 않는다.

### Yahoo

일반 개발자가 사용할 수 있는 공식 뉴스 검색·아카이브 API를 확인하지 못했다. Yahoo Finance의 비공개 엔드포인트나 역공학 API는 운영 데이터 소스로 사용하지 않는다.

### GDELT

- 정확한 시작·종료시각을 지정할 수 있다.
- 무료 DOC API는 검색 범위와 결과 수 제한이 있다.
- 한국어 경제 뉴스 검색 품질과 출처 커버리지가 StockEcho의 주 데이터 소스로 사용하기에 부족했다.
- 보조 실험 후보로만 남긴다.

### NewsAPI

- `from`, `to` 날짜 검색과 유료 5년 아카이브를 지원한다.
- 무료 플랜은 개발 전용이고 최근 데이터만 제공한다.
- 운영용 과거 검색은 유료이며 한국어 커버리지가 핵심 요구에 맞지 않는다.

### 검색결과 페이지 크롤링

NAVER와 Google 검색결과 HTML, 내부 비공개 API, 로그인·캡차·유료벽을 크롤링하거나 우회하지 않는다.

공식 API 결과에서 제공하는 제목, 요약, 발행시각, URL만 사용한다. 언론사 본문 전체를 별도로 크롤링하거나 저장하지 않는다.

언론사가 공식 RSS를 제공하고 이용조건이 명확한 경우에는 향후 allowlist 기반 보조 source adapter로 검토할 수 있다.

## 8. OpenDART의 역할

OpenDART는 뉴스 대체재가 아니라 향후 추가할 **공식 기업 사건 데이터 소스**다.

2015년 이후 회사별 기간을 지정해 다음과 같은 공시를 받을 수 있다.

- 유상·무상증자와 감자
- 합병·분할
- 영업 양수·양도와 영업정지
- 유형자산·타법인 지분 취득과 처분
- 전환사채 등 자금조달
- 자기주식
- 소송, 부도, 회생절차
- 임원·주요주주 지분 변화
- 분기·반기·연간 재무제표

`노조 파업`, `원재료 가격 상승`, `제품 흥행`, `정책 가능성`처럼 공시되지 않는 뉴스성 이슈는 잡지 못한다. MVP 완료 후 `source=dart` Event를 추가해 NAVER 뉴스 Event와 함께 검색하는 확장안으로 남긴다.

## 9. Supabase 용량 원칙

2026년 7월 23일 확인한 개발 DB 상태는 다음과 같았다.

```text
지원 종목: 21
news_articles: 약 9,572
article_stocks: 약 21,310
전체 DB: 약 34.5 MB
```

Free 프로젝트는 DB 크기 여유가 작으므로 검색 API가 반환한 모든 결과를 무조건 영구 저장하지 않는다.

```text
eligible
→ 분석과 과거 Event 검색에 영구 사용

candidate
→ 품질 검토와 재평가용으로 저장

rejected
→ 장기 보관 정책을 별도로 두고 분석에서는 제외
```

동일 기사는 URL, 정규화 제목·발행일·언론사, 콘텐츠 해시를 이용해 중복 제거한다. Supabase의 기사 수보다 기사-검색어, 기사-종목 관계가 더 빠르게 증가할 수 있으므로 두 관계 테이블의 크기도 함께 모니터링한다.

## 10. 법적·운영 안전선

- NAVER API HUB의 공식 endpoint와 발급받은 Key만 사용한다.
- Key는 `.env`에 두고 로그·테스트 출력·문서에 기록하지 않는다.
- API가 제공한 제목·요약·발행시각·URL 범위 안에서 사용한다.
- 원문 기사는 출처 링크로 이동시킨다.
- 검색결과를 전체 뉴스 아카이브인 것처럼 오인시키지 않는다.
- 유료·상업 서비스 공개 전 장기 저장과 파생 분석 용도를 NAVER에 서면 문의한다.
- 403, 429, 한도 초과 응답을 우회하지 않는다.
- API 한도와 RPS 제한보다 낮은 애플리케이션 자체 호출 예산을 둔다.

## 11. 다음 구현 작업

1. 현재 주요 이슈 카드의 `과거 유사 이슈 보기` 버튼에서 실제 Topic/Event 정보를 전달한다.
2. `backfill_issue_news`의 결과를 Supabase 기반 서비스 계층과 연결한다.
3. 검색 결과를 기사 묶음이 아닌 과거 Event 단위로 정제한다.
4. 현재 Event와 과거 Event의 유사도와 채택 근거를 저장한다.
5. 각 과거 Event에 주가 D+1·D+5·D+15·D+30 수익률을 연결한다.
6. 로딩, 결과 없음, 부분 데이터, 오류 상태를 화면에 구현한다.
7. 데이터 출처와 분석 범위 안내를 표시한다.
8. 대표 테스트 Topic으로 정상·부족·중복·휴장일 케이스를 검증한다.

## 12. MVP 완료 조건

- 버튼 클릭 시 목업이 아닌 실제 NAVER 검색 또는 캐시 결과가 표시된다.
- 동일 요청은 캐시를 사용해 API를 반복 호출하지 않는다.
- 현재 Event 이후 기사는 과거 후보에 포함되지 않는다.
- 관련도와 유사도 최저 기준을 통과한 Event만 표시된다.
- 과거 Event는 대표 기사, 날짜, 출처 수, 유사 근거를 가진다.
- D+1·D+5·D+15·D+30은 달력일이 아니라 거래일 기준이다.
- 데이터가 부족하면 명시적인 부족 상태를 보여준다.
- 원문 링크와 데이터 범위 안내가 표시된다.
- 기존 주요 이슈 화면과 분석 worker 흐름이 깨지지 않는다.

## 13. 한 줄 요약

> StockEcho MVP는 NAVER API HUB를 이용해 오늘부터 뉴스를 축적하고, 버튼 클릭 시 현재 이슈의 핵심어로 과거 후보를 제한적으로 검색·캐시한 뒤, 신뢰할 수 있는 Event만 거래일 수익률과 함께 보여준다.
