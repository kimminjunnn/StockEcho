# StockEcho 제품 요구사항 문서(PRD)

> 작성일: 2026-07-21
> 상태: MVP 구현 기준
> 제품명: StockEcho
> 핵심 기능: Risk Replay

## 1. 제품 정의

StockEcho는 사용자가 보유한 국내 주식의 최근 뉴스·공시를 사건 단위로 정리하고, 과거 유사 사건 이후의 실제 주가 반응과 포트폴리오 비중을 결합해 이해하기 쉬운 영향 시나리오를 보여주는 투자 의사결정 보조 서비스다.

### 핵심 사용자 질문

1. 내가 가진 종목에 지금 무슨 일이 생겼는가?
2. 이 기사가 정말 해당 종목과 관련 있는가?
3. 비슷한 사건 뒤 주가는 실제로 어떻게 움직였는가?
4. 내 포트폴리오에서 어느 종목이 가장 크게 영향을 받는가?
5. 이 판단은 어떤 뉴스·공시·가격 데이터에 근거하는가?

### 핵심 가치

- 수많은 기사를 나열하지 않고 종목과의 관계를 설명한다.
- 예측값만 제시하지 않고 과거 사건과 실제 가격 반응을 근거로 보여준다.
- 데이터 기준시각과 신뢰도를 명확히 표시한다.
- 투자 권유가 아니라 사용자의 판단을 돕는 정보 도구로 동작한다.

## 2. 제품 원칙

1. **근거 우선**: 모든 사건과 영향 결과는 출처 문서와 가격 데이터로 역추적할 수 있어야 한다.
2. **정확도 우선**: 많은 기사를 보여주는 것보다 관련 없는 기사를 제외하는 것이 중요하다.
3. **과거와 예측 구분**: 실제 관측 수익률과 모델 추정치를 같은 표현으로 섞지 않는다.
4. **최신성 표시**: 모든 데이터에 `asOf`와 필요 시 `stale`을 표시한다.
5. **부분 실패 허용**: 외부 API가 실패해도 마지막 정상 데이터로 화면이 열린다.
6. **설명 가능한 MVP**: 복잡한 모델보다 검증 가능한 규칙 기반 기준선을 먼저 완성한다.

## 3. 목표 사용자

### 주요 사용자

- 국내 대형주 1~5개를 보유한 개인 투자자
- 뉴스는 보지만 자신의 포트폴리오 영향으로 연결하기 어려운 사용자
- 숫자와 근거 문서를 함께 확인하려는 사용자

### 보조 사용자

- 금융·데이터 분석을 공부하는 학생
- 사건 기반 시장 반응을 탐색하는 리서치 입문자

## 4. MVP 범위

### 4.1 지원 종목

- KOSPI 인기 대형주 20개
- 종목 목록은 [백엔드·데이터 수집 아키텍처 결정안](./MVP_백엔드_데이터_수집_아키텍처_결정안.md)을 따른다.
- 사용자는 1~5개 종목을 선택한다.
- 각 비중은 0보다 크고 합계는 100%여야 한다.
- 종목 지원 여부는 DB에서 관리하고 분기마다 검토한다.

### 4.2 데이터

- KIS 현재가와 일봉
- NAVER API HUB 뉴스 제목·요약·링크
- OpenDART 공시 메타데이터·원문 링크
- 기사-종목 관련도와 판정 근거
- 사건 이후 1·5·20 거래일 가격 반응

### 4.3 사용자 상태

- MVP 로그인 없음
- 보유 종목·비중은 브라우저 `localStorage`에 저장
- 분석 요청 시에만 Next.js 서버로 전달
- 사용자 포트폴리오는 서버 DB에 영구 저장하지 않음

### 4.4 P0 기능

1. 지원 종목 검색·선택·삭제
2. 포트폴리오 비중 입력과 검증
3. 종목별 최근 관련 뉴스·공시
4. 기사 관련 유형과 근거 표시
5. 최신 사건 요약
6. 과거 사건 이후 1·5·20일 가격 반응
7. 비중을 반영한 포트폴리오 영향 시나리오
8. 데이터 기준시각·신뢰도·출처 표시
9. 외부 API 실패 시 마지막 정상 데이터 표시

### 4.5 P1 기능

- 지도학습 기반 기사 관련도·방향 분류
- 유사 사건 검색과 근거 기반 요약
- 비중 변경 시뮬레이션
- 공통 산업 위험 노출 지도
- 사용자 로그인과 포트폴리오 동기화

### 4.6 Non-Goals

- 자동매매와 주문 실행
- 특정 종목 매수·매도 추천
- 수익률 보장 또는 단정적 가격 예측
- 초 단위 실시간 체결 시스템
- 뉴스 본문 무제한 크롤링·복제
- 전 종목 지원
- MVP부터 BERTopic·pgvector·대형 모델을 필수로 운영

## 5. 핵심 사용자 흐름

```text
서비스 진입
→ 지원 종목 1~5개 선택
→ 비중 입력
→ 포트폴리오 분석 요청
→ 전체 영향 요약 확인
→ 가장 영향이 큰 종목 확인
→ 최근 사건·관련 기사·공시 확인
→ 과거 사건 이후 가격 반응 확인
→ 원문과 데이터 기준시각 확인
```

### 예외 흐름

- 비중 합계가 100%가 아니면 분석 버튼을 비활성화하고 수정 안내
- 지원하지 않는 종목 코드는 서버에서 거부
- 현재가 실패 시 마지막 일봉 종가를 `EOD_FALLBACK`으로 표시
- 최신 뉴스 수집이 오래됐으면 `stale` 배지와 마지막 성공시각 표시
- 관련 사건이 없으면 억지 분석 대신 데이터 부족 상태 표시
- 과거 비교 표본이 부족하면 표본 수와 낮은 신뢰도를 표시

## 6. 기능 요구사항

### FR-01. 종목·포트폴리오 입력

- 지원 20종목에서 검색·선택할 수 있다.
- 최대 5개까지 선택할 수 있다.
- 중복 종목을 추가할 수 없다.
- 비중 합계 100%를 실시간 검증한다.
- 새로고침 후에도 localStorage에서 복원한다.

### FR-02. 포트폴리오 종합 진단

- 선택 종목의 최신 사건 수와 영향도를 요약한다.
- 비중을 반영해 영향 기여도가 큰 종목 순으로 정렬한다.
- 결과가 실제 과거 반응인지 모델 추정인지 명확히 구분한다.
- 투자 권유가 아님을 표시한다.

### FR-03. 종목별 최근 뉴스·공시

- 제목, 요약, 발행시각, 출처, 원문 링크를 표시한다.
- 기사-종목 관계를 `direct`, `product`, `industry`로 표시한다.
- 관련도 confidence와 사람이 이해할 수 있는 evidence를 제공한다.
- 동일 기사는 한 번만 표시한다.
- 단순 주가 나열·무관 기사 등 낮은 품질 결과를 제외한다.

### FR-04. 사건 상세

- 대표 사건명, 사건일, 방향, 신뢰도를 표시한다.
- 연결된 뉴스와 공시를 함께 보여준다.
- 공시가 있으면 공식 근거로 우선 표시한다.
- 사건 생성 규칙 또는 모델 버전을 추적 가능하게 저장한다.

### FR-05. Risk Replay

- 사건 기준 실제 거래일의 1·5·20일 수익률을 표시한다.
- 휴장일은 거래일 계산에서 제외한다.
- 장 마감 후 사건은 다음 거래일을 기준으로 삼는다.
- 과거 사건 표본 수와 분포를 표시한다.
- 표본이 부족하면 결과를 생성하지 않거나 낮은 신뢰도로 표시한다.

### FR-06. 포트폴리오 영향

- 종목별 사건 영향을 사용자 비중으로 가중한다.
- 종목별 기여도를 분리해서 보여준다.
- 사용자가 비중을 바꾸면 결과를 다시 계산한다.
- 결과를 예상 수익률이 아니라 과거 기반 영향 시나리오로 표현한다.

### FR-07. 근거와 최신성

- 모든 화면에 데이터 기준시각 `asOf`를 표시한다.
- 데이터가 오래됐으면 `stale`을 표시한다.
- 사건에서 원문 뉴스·공시로 이동할 수 있다.
- 가격 데이터 원천과 가격 유형을 표시한다.

## 8. 데이터·분석 요구사항

세부 데이터 구조는 [MVP 데이터 모델링 결정안](./MVP_데이터_모델링_결정안.md)을 따른다.

### P0 분석

- URL·content hash 기반 중복 제거
- 규칙 기반 기사 관련도
- Kiwi·TF-IDF 키워드 후보
- 규칙 기반 사건 연결
- 실제 사건 후 가격 반응
- 비중 가중 포트폴리오 영향

### 모델 도입 조건

- 가격 반응 등 재현 가능한 방식으로 target을 생성할 수 있다.
- 시간순 validation·test로 평가한다.
- 규칙 기반 기준선보다 의미 있게 개선된다.
- 결과에 dataset·model·rule version을 남긴다.
- 오류 사례를 입력 데이터와 규칙·모델 버전으로 재현할 수 있다.

BERTopic, SentenceTransformer, pgvector는 P0 출시 조건이 아니다.

## 9. 서비스 구조

세부 운영 구조는 [백엔드·데이터 수집 아키텍처 결정안](./MVP_백엔드_데이터_수집_아키텍처_결정안.md)을 따른다.

```text
Next.js on Lightsail
├─ Server Components → DAL → Supabase
├─ 브라우저 비동기 요청용 Route Handlers
└─ KIS 현재가 5~10초 메모리 캐시

Lightsail Python jobs
├─ NAVER·KIS·OpenDART 증분 수집
├─ 정규화·관련도·키워드 처리
└─ Supabase idempotent upsert

Supabase
├─ PostgreSQL 정제 데이터
└─ Private Storage 선별 Raw·데이터셋·모델
```

MVP에는 FastAPI를 두지 않는다. Next.js Server Component는 자기 Route Handler를 다시 호출하지 않고 DAL을 직접 사용한다.

## 10. API 요구사항

Route Handler는 브라우저 비동기 요청에만 사용한다.

| Method | Path | 목적 |
|---|---|---|
| `GET` | `/api/stocks/[stockCode]/price` | 현재가 갱신 |
| `GET` | `/api/stocks/[stockCode]/news` | 뉴스 페이지네이션 |
| `POST` | `/api/portfolio/analyze` | localStorage 입력 분석 |
| `GET` | `/api/health` | 서비스 상태 |

서버 렌더링 페이지의 초기 데이터는 DAL을 직접 조회한다.

### 분석 입력 검증

- 활성 지원 종목 allowlist
- 1~5개 종목
- 각 비중 0 초과 100 이하
- 합계 100
- 알 수 없는 필드 거부 또는 제거
- Nginx `limit_req` 기반 rate limit

## 11. 비기능 요구사항

### 성능

- 저장된 데이터 기반 초기 결과를 2초 이내 표시하는 것을 목표로 한다.
- 현재가 외부 호출은 사용자 화면 전체 렌더링을 차단하지 않는다.
- 종목·기사 목록은 필요한 필드만 조회하고 페이지네이션한다.

### 신뢰성

- 수집과 사용자 요청을 분리한다.
- 모든 저장은 idempotent upsert다.
- 수집 작업은 중복 실행되지 않는다.
- 외부 API 실패 시 마지막 정상 데이터로 fallback한다.

### 보안

- 브라우저는 외부 API와 Supabase Secret에 접근하지 않는다.
- Secret은 서버 전용 환경 변수에 둔다.
- Colab에 Supabase Secret을 넣지 않는다.
- 오류 응답과 로그에 인증값을 남기지 않는다.

### 접근성·반응형

- 모바일에서도 핵심 결과와 출처 링크를 확인할 수 있다.
- 색상만으로 위험·상태를 구분하지 않는다.
- 키보드로 주요 입력과 탐색이 가능해야 한다.

### 금융 안전 문구

- 본 서비스는 투자 판단을 보조하는 정보 도구다.
- 결과는 매수·매도 추천이나 수익 보장이 아니다.
- 과거 수익률은 미래 수익률을 보장하지 않는다.

## 12. 성공 지표

### 제품 지표

- 사용자가 1~5개 종목과 비중을 입력해 결과까지 완료하는 비율
- 종목 상세·원문 근거 클릭률
- 분석 완료까지 걸린 시간
- unsupported·validation 오류율

### 데이터 품질 지표

- 직접 관련 기사 precision
- 동일 기사 중복 노출률
- 종목별 신규·무시 기사 수
- 마지막 정상 수집 이후 경과시간
- 기사-종목 연결 중 evidence 누락률

### 운영 지표

- NAVER 일일 호출량 정상 시 3,000회 이내
- 장중 직접 뉴스 최신성 90분 이내
- 수집 작업 연속 실패 횟수
- Raw Storage와 서버 디스크 사용량

## 13. 구현 순서

### Phase 1. 핵심 5종목 데이터

1. 삼성전자 최신순 증분 수집
2. 정규화·중복 제거·공통 관련도 규칙

## 14. MVP 완료 기준

- 지원 20종목을 선택하고 1~5개 포트폴리오를 구성할 수 있다.
- 최근 뉴스·공시·일봉이 회사 master로 연결된다.
- 기사마다 종목 관계·점수·근거를 설명할 수 있다.
- 동일 기사와 관계가 중복되지 않는다.
- 최신 사건과 실제 1·5·20일 가격 반응을 확인할 수 있다.
- 포트폴리오 비중을 반영한 영향 시나리오가 계산된다.
- 외부 API 실패 시 마지막 정상 데이터와 `asOf/stale`이 표시된다.
- Secret이 브라우저·Git·로그·Colab에 노출되지 않는다.
- 학습 모델 없이도 핵심 사용자 흐름이 끝까지 동작한다.
